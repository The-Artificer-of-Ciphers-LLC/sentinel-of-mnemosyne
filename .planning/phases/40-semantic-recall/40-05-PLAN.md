<objective>
Close the protection half of the Phase-40 UAT blocker, tightened per cross-AI review (40-REVIEWS.md):
the sweeper RELOCATED the operator-critical `sentinel/persona.md` because nothing in the vault treated
`sentinel/` as never-movable. `sentinel/` is NOT in `sweep_skip_prefixes` — and even if it were,
skip-prefix only governs whether a note is PROCESSED, not whether it may be MOVED. The persona crash-looped
boot at `composition.py:424` (`'sentinel/persona.md missing from Vault'`).

This plan adds a Vault-seam-level protected-namespace guard so operator-critical paths can NEVER be
relocated or trashed — by ANY caller (startup rebuild, admin sweep, force_reclassify sweep, or future code).
The guard lives on `ObsidianVault.relocate` and `ObsidianVault.move_to_trash` — the two mutation primitives
that physically move files — so it is enforced regardless of which decision logic invoked them. Per review:
1. The default protected set is ENUMERATED explicitly (concern 5), not left implicitly `sentinel/`-only.
2. Destination protection is added (concern 6): relocating content INTO a protected namespace is refused,
   not just out of it.
3. Plan hygiene (concern 7): `app/errors.py` is in `files_modified`; the test file is the REAL
   `tests/test_obsidian_vault.py` (not a non-existent `test_vault.py`); acceptance criteria use
   behavior/spy assertions, not brittle grep-only counts.

Relationship to 40-04 (concern 8): this plan adds the PRIMITIVE-LEVEL guard (raises ProtectedPathError);
40-04 OWNS `vault_sweeper.py` and ensures every destructive sweep branch CATCHES ProtectedPathError,
records it, and continues. The two plans run in the SAME wave with ZERO file overlap — to keep that true,
ALL tests in THIS plan live in `tests/test_obsidian_vault.py` (the primitive's test module) and this plan
does NOT touch `tests/test_vault_sweeper.py`. The cross-plan INTEGRATION test (a real sweep over
sentinel/persona.md that records the refusal and continues across all three destructive branches) lives in
40-04, which owns vault_sweeper.py. This plan's tests assert only that the PRIMITIVE raises and the file is
untouched — which is sufficient because FakeVault delegates relocate/move_to_trash to the real ObsidianVault
method bodies, so the guard is exercised through both adapters.

Purpose: defense-in-depth. Even if 40-04's startup/runtime guards were bypassed by a future regression,
a protected file physically cannot be relocated or trashed, and arbitrary content cannot be moved into a
protected namespace.
Output: `PROTECTED_NAMESPACES` set + `is_protected_path()` predicate + `ProtectedPathError`, enforced on
both source and destination inside `relocate`/`move_to_trash`, with an enumerated env-overridable default.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sweep decision logic → vault file moves | Any classifier-driven move crosses into physical relocation/trash. Operator-critical paths must never cross this boundary as a SOURCE. |
| arbitrary content → protected namespace | A future caller could relocate non-operator content INTO sentinel/ (namespace poisoning). The destination must also be guarded. |
| future code paths → relocate/trash primitives | The guard lives on the primitive (Vault seam) so ANY future caller inherits the protection, not just today's sweeper. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-40-17 | Tampering / Data-loss | relocation/trash of operator-critical protected files (SOURCE) | high | mitigate | `is_protected_path(src)` guard is the first statement of `relocate`/`move_to_trash`; raises `ProtectedPathError` before any copy/delete (Task 2 acceptance: source untouched, delete un-called). |
| T-40-18 | Tampering / Bypass | a future caller bypassing sweeper-level guards | medium | mitigate | Guard lives on the Vault primitive, not in sweep decision logic — every caller inherits it. FakeVault delegates to ObsidianVault so tests exercise the same guard. |
| T-40-26 | Tampering / Namespace poisoning | relocating arbitrary content INTO a protected namespace (DESTINATION) | medium | mitigate | `is_protected_path(dst)` guard in `relocate` raises before any write into a protected namespace (Task 2 concern-6 test: nothing written under sentinel/). |
| T-40-19 | Elevation of Privilege / Config | over-broad or operator-misconfigured protected set | low | accept | `protected_namespaces` is env-overridable but defaults to an enumerated safe set. Over-protection only blocks moves (fail-safe), never enables one; acceptable. |
