<objective>
Close the root-cause half of the Phase-40 UAT blocker: the first boot's "non-blocking startup index
rebuild" (D-06) called the FULL destructive `run_sweep` (walk → classify → embed → dedup → RELOCATE)
instead of an index-only routine. With the embedding model not yet loaded on first boot, classifications
were degraded, and the sweeper RELOCATED `sentinel/persona.md` → `learning/persona/`, crash-looping the
next boot at `composition.py:424`.

This plan:
1. Adds a NEW index-only routine `rebuild_embedding_index()` that walks the vault, (re)embeds bodies,
   and writes `ops/sweeps/embedding-index.json` via the existing `_emit_embedding_index` helper — with
   ZERO classify / relocate / trash / de-dup calls.
2. Rewires the startup rebuild in `composition.py` to call `rebuild_embedding_index()` instead of
   `run_sweep()`.
3. Adds an embeddings-degraded abort guard to `run_sweep` so even a manual/admin full sweep refuses to
   relocate, trash, or de-dup when the embedding model is unavailable (acting on bad classifications is
   the failure mode that mis-classified the persona).

Purpose: a cold start must become semantically searchable WITHOUT ever moving a vault file, and a sweep
must never act destructively on degraded model output.
Output: `rebuild_embedding_index()` routine, degraded-sweep abort guard, rewired startup path, regression
tests proving `run_sweep` is never called at boot and that a degraded sweep moves nothing.

This plan does NOT regress the 4 shipped success criteria (RetrievalStrategy seam, SemanticRecall reading
the sidecar, embedding_model mismatch skip, RRF k≈60 merge) — it only changes WHICH routine boot calls and
adds a fail-safe gate; the index-write path (`_emit_embedding_index`) is reused verbatim.
</objective>
