<objective>
Make the vault sweeper the producer of the semantic-recall index. The sweeper already
computes per-note embeddings and writes `embedding_b64` into note frontmatter; this plan
adds the three new responsibilities that turn those embeddings into queryable retrieval
data (MEM-05, producer side):

1. Emit a sidecar index `ops/sweeps/embedding-index.json` keyed by note path, each entry
   carrying `{ embedding_b64, embedding_model, content_hash }` (D-03, D-04, D-07).
2. Rebuild the index incrementally by content-hash — carry unchanged entries forward,
   re-embed changed/new notes, prune entries for trashed/deleted notes (D-05).
3. Prepend the nomic `search_document:` prefix to note bodies before embedding so that
   document and query embeddings share the same instruction space (RESEARCH Pattern 6).

Persistence goes THROUGH the Vault seam (`vault.write_note()`) — the vault is REST-only,
so there is NO `tempfile`/`os.replace`/local filesystem write (D-08 REVISED, A6).

Purpose: Without a sweeper-maintained index, SemanticRecall (Plan 02) has nothing to read.
Output: Updated `vault_sweeper.py` (`_emit_embedding_index`, `_content_hash`, the nomic
document prefix constant, the index-path constant) + new tests in `test_vault_sweeper.py`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| note content → embedder | Untrusted user-authored vault note bodies are read, prefixed, and embedded |
| sweeper → Vault (REST) | The index JSON is serialized and PUT to the vault via Obsidian REST |
| prior index → sweeper | The previously-written `embedding-index.json` is read back and parsed as JSON (untrusted-at-rest input) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-01 | Tampering | `_emit_embedding_index` reading prior index JSON | mitigate | `json.loads` wrapped in try/except → `{}` on any parse failure (malformed/oversized/injected content cannot crash the sweep); a corrupt index self-heals on next sweep |
| T-40-02 | Denial of Service | sweeper full re-embed cost | accept | The one-time prefix-driven full re-embed is bounded by vault size (personal scale); steady state is incremental via content-hash. No external trigger amplification (sweep is admin-gated at the route layer — see Plan 03) |
| T-40-03 | Tampering | note paths used as index keys | mitigate | Keys are vault-relative paths already produced by the sweeper's own `walk_vault`/classify; no path traversal vector — the sweeper never opens local files, all I/O is via the Vault seam |
| T-40-04 | Information disclosure | embedding_b64 vectors in a JSON sidecar under ops/ | accept | The index lives under `ops/` (already excluded from recall candidates and from user-facing surfaces); it contains only derived embeddings, no new secrets |
| T-40-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages added this phase (RESEARCH Package Legitimacy Audit: none); nothing to install |
</threat_model>
